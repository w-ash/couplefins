### v0.5.0: Split Editing (Vertical Slice 1)
**Goal**: Edit split percentages on individual transactions or in bulk, directly from the transactions page.
**Context**: Splits are currently set only during CSV import via Monarch `sXX` tags. The couple needs to fix incorrect splits or override defaults without re-uploading the entire CSV.
**What this unlocks**: Direct control over split allocation post-import.
**Key tech choices**: Single backend endpoint handles both individual and bulk updates (list of one vs many). Expandable row detail with live share preview. Finalization guards prevent edits to locked months.

### v0.5.1: Transaction Field Editing + Audit Log
**Goal**: Edit date, amount, category, and tags on individual transactions, with a full audit log of every change.
**Context**: Splits were the first editable field (v0.5.0). The couple also needs to fix dates, amounts, categories, and tags after import — without re-uploading the CSV. Every edit must be logged (before/after + timestamp) for trust and auditability.
**What this unlocks**: Full post-import transaction editing with change tracking.
**Key tech choices**: `original_date`/`original_amount` fields preserve natural key for CSV dedup. `TransactionEdit` entity logs each field change. Unified `TransactionEditor` replaces `SplitEditor` in expanded row. Bulk split editing stays on existing endpoint.

### v0.5.5: Transaction Search, Filtering & Date Range
**Goal**: Replace the month selector with a date range picker, and add search + faceted filters so the couple can quickly find and act on specific transactions during solo prep.
**Context**: With 100-200 transactions per month, scanning a date-sorted list isn't efficient. The month selector is too rigid for cross-month searches. The couple needs to find specific merchants, filter by tags or categories, and drill down by payer or amount — all without leaving the transactions page.
**What this unlocks**: Fast transaction lookup. Flexible date ranges beyond calendar months. Filter state in URLs for sharing.
**Key tech choices**: Date range is the only server-side filter (API change). Everything else is client-side on already-loaded data. `useDateRange()` hook replaces `useMonthYear()` on Transactions page only. Grouped multi-select for categories built on shadcn Combobox + Popover primitives. Active filters shown as dismissible pills. OR logic within filter groups, AND across groups.

#### Note: Settlement & Finalization

The finalization banner currently lives on the Transactions page. With date ranges spanning multiple months, per-month finalization status doesn't map cleanly. More fundamentally, finalization ("lock the month") is part of the settlement flow, not the transaction review flow. v0.6.0 addresses this by giving settlement its own dedicated page and moving finalization there.
