### v0.5.0: Split Editing (Vertical Slice 1)
**Goal**: Edit split percentages on individual transactions or in bulk, directly from the transactions page.
**Context**: Splits are currently set only during CSV import via Monarch `sXX` tags. The couple needs to fix incorrect splits or override defaults without re-uploading the entire CSV.
**What this unlocks**: Direct control over split allocation post-import.
**Key tech choices**: Single backend endpoint handles both individual and bulk updates (list of one vs many). Expandable row detail with live share preview. Finalization guards prevent edits to locked months.

### v0.5.1: Transaction Field Editing + Audit Log
**Goal**: Edit date, amount, category, and tags on individual transactions, with a full audit log of every change.
**Context**: Splits were the first editable field (v0.5.0). The couple also needs to fix dates, amounts, categories, and tags after import — without re-uploading the CSV. Every edit must be logged (before/after + timestamp) for trust and auditability.
**What this unlocks**: Full post-import transaction editing with change tracking.
**Key tech choices**: `original_date`/`original_amount` fields preserve natural key for CSV dedup. `TransactionEdit` entity logs each field change. Unified `TransactionEditor` replaces `SplitEditor` in expanded row. Bulk split editing stays on existing endpoint.

#### Domain + Dedup Foundation

- [x] **Extend Transaction entity + update dedup**
    - Effort: S
    - Status: Completed (2026-03-12)
    - Dependencies: v0.5.0
    - What: Add `original_date: date | None` and `original_amount: Decimal | None` to `Transaction`. Update `natural_key()` to use originals when set.
    - Why: Date and amount are part of the dedup natural key. Editing them would break CSV re-upload matching — storing originals preserves idempotent re-import.
    - Notes:
        - `src/domain/entities/transaction.py` — add two nullable fields with `= None` defaults
        - `src/domain/dedup.py` — `natural_key()` uses `tx.original_date or tx.date` and `tx.original_amount or tx.amount`
        - `_MUTABLE_FIELDS` unchanged — date/amount are UI-mutable, not CSV-mutable
        - Update dedup tests to cover original-value fallback

- [x] **TransactionEdit entity + repository protocol**
    - Effort: S
    - Status: Completed (2026-03-12)
    - Dependencies: v0.5.0
    - What: New `TransactionEdit` frozen attrs entity. New `TransactionEditRepositoryProtocol` with `save_batch()` and `get_by_transaction_id()`. Add to `UnitOfWorkProtocol`.
    - Why: Audit trail for every field change on a transaction
    - Notes:
        - `TransactionEdit`: id, transaction_id, field_name, old_value, new_value, edited_at
        - No `edited_by_person_id` — two-person trusted-device app, timestamp is sufficient

### v0.5.5: Transaction Search & Filtering
**Goal**: During solo prep, quickly find transactions to review or fix.
**Context**: With 100-200 transactions per month, scanning a date-sorted list isn't efficient when looking for a specific merchant, category, or large amount. All data is already client-side (loaded via reconciliation query), so no API changes needed.
**What this unlocks**: Fast transaction lookup during solo prep. Filter state persisted in URL for sharing.

#### Search

- [ ] **Text search bar**
    - Effort: S
    - Dependencies: v0.5.4
    - What: Search input above the transaction table. Case-insensitive substring match across merchant, category, notes, and tags. Debounced input (150ms). Clear button. Result count: "Showing 12 of 87 transactions".
    - Why: Fastest way to find a specific transaction.

#### Filters

- [ ] **Payer, category group, and amount filters**
    - Effort: M
    - Dependencies: "Text search bar"
    - What: Compact filter row below search. Payer: toggle pills for each person. Category group: dropdown or pill set (from existing `categoryGroups` map). Amount range: optional min/max inputs. Active filters shown as dismissible pills. Client-side filtering — `sorted` memo becomes `filtered` memo chaining search → filters → sort.
    - Why: Drill down to specific subsets during review.

#### Sort Controls

- [ ] **Clickable column header sorting**
    - Effort: S
    - Dependencies: v0.5.4
    - What: Replace fixed date-desc sort with clickable column headers: Date (default desc), Merchant (alpha), Amount (largest first), Category group. Small chevron indicator on active sort column.
    - Why: Currently hard-coded sort order. Users may want to find largest transactions or scan by category.

#### URL Persistence

- [ ] **Persist filters in URL search params**
    - Effort: S
    - Dependencies: "Payer, category group, and amount filters", "Clickable column header sorting"
    - What: Store search, filters, and sort as URL params (`?q=dinner&payer=uuid&group=Food&sort=amount`). Refreshing preserves filters. Follows existing `useMonthYear()` pattern.
    - Why: Don't lose your filter state on refresh. Can share a filtered view URL with partner.
