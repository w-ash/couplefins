### v0.6.0: Settlement Page & Recording
**Goal**: Give settlement its own dedicated page and let the couple record payments to close out a month's balance.
**Context**: Settlement is the core "together moment" — the reason the couple opens the app side by side. Currently the settlement card is duplicated on Dashboard and Transactions pages, and finalization controls are embedded in the Transactions page. Settlement deserves a focused experience. The reconciliation engine computes "Alice owes Bob $147.50" but there's no way to record that the payment actually happened.
**What this unlocks**: Clear separation between solo prep (Transactions), overview (Dashboard), and the together session (Settlement). Completing the core loop — upload → reconcile → settle → finalize becomes a tracked end-to-end workflow on a dedicated page.
**Key tech choices**: New `/settle` route with month selector (settlement is inherently monthly). `Settlement` entity with payment method enum. Recording a payment is not guarded by finalization (it's documenting what happened), but deleting one is. Settlement amount pre-filled from computed balance. Finalization controls move here from Transactions page.

#### Settlement Domain + Persistence

- [ ] **Settlement entity + repository**
    - Effort: S
    - Dependencies: v0.5.5
    - What: New `Settlement` frozen attrs entity: id, year, month, amount, from_person_id, to_person_id, method (Venmo/Zelle/Cash/Other), settled_at, notes. New `SettlementRepositoryProtocol` with `save()`, `get_by_period()`, `get_by_year()`, `delete()`. Alembic migration for `settlements` table. Wire into `UnitOfWorkProtocol`.
    - Why: Persistence foundation for the settlement workflow

#### Settlement Use Cases

- [ ] **RecordSettlement + GetSettlements + DeleteSettlement use cases**
    - Effort: M
    - Dependencies: "Settlement entity + repository"
    - What: `RecordSettlementUseCase` creates a settlement record (validates period exists, amount positive, person IDs exist). `GetSettlementsUseCase` fetches settlements for a period. `DeleteSettlementUseCase` removes a mistaken entry (guarded by finalization).
    - Why: Core backend logic for settlement tracking
    - Notes:
        - Recording is not finalization-guarded — paying a debt is documenting reality
        - Deletion is finalization-guarded — can't un-record a payment on a locked month

#### Settlement Page + API

- [ ] **Settlement endpoints**
    - Effort: S
    - Dependencies: "RecordSettlement + GetSettlements + DeleteSettlement use cases"
    - What: `POST /api/v1/settlements` to record, `GET /api/v1/settlements?year=X&month=Y` to fetch, `DELETE /api/v1/settlements/{id}` to remove.
    - Why: HTTP interface for settlement use cases

- [ ] **Dedicated settlement page**
    - Effort: M
    - Dependencies: "Settlement endpoints"
    - What: New `/settle` route. Month selector (settlement is inherently monthly). Shows settlement amount for the selected month. Category group spending breakdown. Recording UI: "Mark as Settled" button → inline form (amount pre-filled, method dropdown, date picker, notes). After recording, shows "Settled on Mar 12 via Venmo" with undo option. Finalization controls (Lock/Unlock Month). Gentle prompt when both uploads are in and month is settled: "Ready to lock this month?"
    - Why: Settlement is the app's core interaction — it shouldn't be a card buried in other pages. Designed for two people looking at the screen together.

- [ ] **Remove settlement from Dashboard and Transactions**
    - Effort: S
    - Dependencies: "Dedicated settlement page"
    - What: Replace settlement cards on Dashboard and Transactions with a CTA linking to `/settle`. Remove finalization banner from Transactions page.
    - Why: Settlement lives in one place now. Other pages point to it.

- [ ] **Sidebar navigation update**
    - Effort: S
    - Dependencies: "Dedicated settlement page"
    - What: Add "Settle Up" nav item to sidebar. Position it prominently — this is what the together session starts with.
    - Why: Settlement needs to be easy to find when the couple sits down together.

### v0.6.1: Settlement History
**Goal**: See payment history over time on the Dashboard.
**Context**: v0.6.0 lets the couple record settlements on a dedicated page. The Dashboard month history table already shows spending — adding settlement status completes the picture.
**What this unlocks**: At-a-glance view of which months are fully resolved vs. still open.

#### Dashboard Enrichment

- [ ] **Month history settlement status**
    - Effort: S
    - Dependencies: v0.6.0
    - What: Enrich `MonthHistoryEntry` with `is_settled: bool` and `settled_at: datetime | None`. Dashboard month history table shows check/pending icon per month. `GetDashboardUseCase` loads settlements alongside existing data.
    - Why: Quick visual scan of which months are done

- [ ] **Running settlement totals**
    - Effort: S
    - Dependencies: v0.6.0
    - What: "Total settled Alice → Bob this year: $X" summary on Dashboard. Computed from `GetSettlementsUseCase` across the year.
    - Why: Year-level context for the couple's payment flow
