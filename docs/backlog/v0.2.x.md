### v0.2.0: Reconciliation Engine + Transactions Page (Vertical Slice 1)
**Goal**: View all shared transactions for a month from both people, see who owes whom.
**Context**: With uploads working, the core value proposition is reconciliation.
**What this unlocks**: Replaces the spreadsheet for monthly reconciliation.
**Key tech choices**: Pure domain logic for reconciliation math, React table for transaction display.

#### Reconciliation Engine Epic

- [ ] **Implement reconciliation domain logic**
    - Effort: M
    - Status: Not Started
    - What: Pure function in src/domain/ that takes two lists of shared transactions (one per person) and computes the reconciliation summary
    - Why: Core business logic — must be pure and thoroughly tested
    - Dependencies: v0.1.x
    - Notes:
        - For each shared transaction: use transaction.payer_person_id to identify the payer and transaction.payer_percentage for the split math
        - payer_share = |amount| × (payer_percentage / 100), other_share = |amount| × ((100 - payer_percentage) / 100)
        - Aggregate: total shared spending, each person's total paid, each person's total owed share, net settlement amount
        - Per-category breakdown with same math
        - Per-category-group breakdown: aggregate per-category numbers into groups using CategoryMapping. Return both per-category and per-category-group breakdowns
        - Accepts category mappings (or a lookup function) as input alongside transaction lists
        - Return a ReconciliationSummary domain entity

- [ ] **Unit tests for reconciliation engine**
    - Effort: M
    - Status: Not Started
    - What: Comprehensive tests for reconciliation math
    - Why: Financial math must be exact
    - Dependencies: "Implement reconciliation domain logic"
    - Notes:
        - Test cases: all 50/50, mixed splits, one person paid everything, equal spending (net zero), single transaction, many transactions, refunds (positive amounts)
        - Test per-category-group aggregation (multiple categories in same group)
        - Test categories with no group mapping (should appear as "Uncategorized" or similar)
        - Use Decimal or careful float handling for money

- [ ] **Reconciliation use case + API endpoint**
    - Effort: M
    - Status: Not Started
    - What: GET /api/v1/reconciliation?year=YYYY&month=MM — returns full reconciliation for a month
    - Why: Serves the reconciliation data to the frontend
    - Dependencies: "Implement reconciliation domain logic"
    - Notes:
        - Use case queries both persons' shared transactions for the given month
        - Calls reconciliation domain logic
        - Use case also fetches category mappings and passes them to the domain logic
        - Returns: transaction list, per-person totals, per-category breakdown, per-category-group breakdown, net settlement

#### Transactions UI Epic

- [ ] **Transactions page**
    - Effort: L
    - Status: Not Started
    - What: React page showing month's reconciliation: transaction table, summary cards, per-category breakdown. Lives at `/transactions` in the sidebar nav.
    - Why: Primary user-facing view — replaces the spreadsheet
    - Dependencies: "Reconciliation use case + API endpoint", v0.1.3 "App shell, navigation & user identity"
    - Notes:

        - Transaction table: date, merchant, category, category group, who paid, amount, split ratio, each person's share

        - Color coding: payer column, amounts

### v0.2.1: Dashboard + Month Navigation (Vertical Slice 2)
**Goal**: A landing page showing current month summary with easy navigation to past months.
**Context**: Users need a quick overview without navigating to the full transactions page.
**What this unlocks**: At-a-glance status for the current month.

#### Dashboard Epic

- [ ] **Dashboard API endpoint**
    - Effort: S
    - Status: Not Started
    - What: GET /api/v1/dashboard — returns current month summary, YTD summary, and list of available months
    - Why: Powers the landing page
    - Dependencies: v0.2.0 "Reconciliation use case + API endpoint"
    - Notes:
        - Current month: total shared spending, net settlement, upload status per person
        - YTD: total shared spending YTD, net settlement running total, budget health summary (on track / over)

- [ ] **Dashboard page**
    - Effort: M
    - Status: Not Started
    - What: Landing page at `/` with current month summary card (who owes whom), quick links to upload/transactions, list of past months. Becomes the default route (replacing `/upload` redirect from v0.1.3).
    - Why: First thing users see — should give instant orientation
    - Dependencies: "Dashboard API endpoint", v0.1.3 "App shell, navigation & user identity"
    - Notes:
        - Summary card: "For March 2026: Alice owes Bob $127.50" (or "All settled!")
        - YTD summary card: total shared spend YTD, net settlement running total, budget health (on track / over)
        - Upload status: "Alice uploaded ✓ / Bob: not yet"
        - Quick action buttons: Upload CSV, View Full Reconciliation
        - Month history list with net amounts

- [ ] **Month navigation component**
    - Effort: S
    - Status: Not Started
    - What: Reusable month/year picker component used across transactions and dashboard pages
    - Why: Consistent navigation across views
    - Dependencies: v0.1.1 "Scaffold React frontend"
