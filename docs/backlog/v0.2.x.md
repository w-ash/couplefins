### v0.2.0: Reconciliation Engine + Transactions Page (Vertical Slice 1)
**Goal**: Answer the question "who owes whom this month?" — the core reason this app exists.
**Context**: Both people upload their Monarch CSVs (v0.1.x). Now they need to see the result: a single settlement number, a full transaction list, and category breakdowns that help them understand where their shared money went.
**What this unlocks**: Replaces the spreadsheet for monthly reconciliation.
**Key tech choices**: Pure domain logic for reconciliation math (Decimal, round-per-transaction). Single fat API endpoint (all data for the page in one call). URL-driven month selection (`?year=&month=`).

#### User Journey

The monthly reconciliation cycle:
1. Month ends. Each person exports their Monarch CSV and uploads it.
2. Both uploads land. They navigate to Transactions for that month.
3. **First question**: "Who owes whom, and how much?" — the hero settlement number.
4. **Second question**: "Is this complete? Did both of us upload?" — upload status banner.
5. **Third question**: "Where did the money go?" — category group breakdown.
6. **Fourth question**: "Do any of these transactions look wrong?" — full transaction table.
7. One person Venmos the other. Done.

#### Information Hierarchy

| Priority | Element | Why |
|---|---|---|
| **Hero** | Settlement card: "Alice owes Bob $127.50" | The single answer they came for |
| **Gating** | Upload status banner (if incomplete) | Prevents "why does the number look wrong?" |
| **Context** | Month selector | Navigate to past months |
| **Summary** | Stats row: total shared, Alice paid X, Bob paid Y | Orientation before detail |
| **Analysis** | Category group breakdown (collapsible → categories) | "Where did the money go?" |
| **Detail** | Transaction table, sorted by date desc | Review individual expenses |
| **Warning** | Unmapped categories (only if present) | Actionable: link to Settings |

#### Edge Cases

| Scenario | What the user sees |
|---|---|
| **Only one person uploaded** | Banner: "Partial data — only Alice has uploaded for March 2026." Summary computes with available data but is clearly labeled incomplete. |
| **Zero shared transactions** | Empty state: "No shared transactions for March 2026." Not an error — maybe everything was personal this month. |
| **No data at all** | Same as zero shared transactions. |
| **Unmapped categories** | Warning banner listing them. Transactions still appear with group column showing "Uncategorized." Don't block reconciliation. |
| **Refunds** (positive amounts) | Math works naturally with signed convention. Shared $20 refund at 50/50 means each person's share of the refund is $10. UI labels positive amounts in teal. |
| **100/0 split** (`s100`) | Payer covers 100%, other owes nothing. Displays "100 / 0" in split column. |
| **0/100 split** (`s0`) | Payer covers 0%, other owes everything. Displays "0 / 100." |
| **Net zero** | Settlement card: "All settled!" |

#### Reconciliation Engine Epic

- [x] **Implement reconciliation domain logic**
    - Effort: M
    - Status: Completed (2026-03-10)
    - What: Pure function `reconcile()` in `src/domain/reconciliation.py` — takes shared transactions, persons, category mappings, and category groups → returns `ReconciliationSummary`
    - Why: Core business logic — must be pure and thoroughly tested
    - Dependencies: v0.1.x
    - Notes:
        - **New file**: `src/domain/reconciliation.py` — value objects + pure reconcile function
        - Value objects (frozen attrs, not persisted entities): `PersonSummary`, `SettlementResult`, `CategoryBreakdown`, `CategoryGroupBreakdown`, `ReconciliationSummary`
        - **Math per transaction**: `abs_amount = abs(tx.amount)`, `payer_share = abs_amount × payer_percentage / 100`, `other_share = abs_amount × (100 - payer_percentage) / 100`
        - **Settlement**: For each person, compute `net_owed = total_share - total_paid`. The person with positive net_owed pays the other. For refunds (positive amount), the payer "overpaid" by a negative amount, reducing what the non-payer owes.
        - **Rounding**: Use `Decimal` throughout. Quantize each per-transaction share to `Decimal("0.01")` with `ROUND_HALF_UP`. Round per-transaction, not in aggregate.
        - **Category grouping**: Build `dict[str, tuple[UUID | None, str]]` lookup from mappings + groups. Unmapped categories get `(None, "Uncategorized")`. Group transactions by category, then roll up into category groups sorted by total amount desc.
        - **Function signature**: `reconcile(transactions: list[Transaction], persons: list[Person], category_mappings: list[CategoryMapping], category_groups: list[CategoryGroup]) -> ReconciliationSummary`
        - **Helpers** (private): `_build_category_lookup`, `_compute_person_summaries`, `_compute_settlement`, `_compute_category_breakdowns`

#### Idempotent Uploads Epic

- [x] **Dedup-aware upload flow**
    - Effort: M
    - Status: Completed (2026-03-10)
    - What: Re-uploading a CSV is idempotent. Transactions are matched by natural key (date, amount, account, original_statement, payer_person_id). New transactions are inserted, unchanged are skipped, and changed transactions (e.g., updated tags or categories in Monarch) are surfaced for review. Month/year inputs removed — dates come from the CSV.
    - Why: Users re-export from Monarch after editing tags/categories. Without dedup, every re-upload creates duplicates.
    - Dependencies: "Reconciliation use case + API endpoint"
    - Notes:
        - Pure domain classification in `src/domain/dedup.py`
        - Upload entity no longer has period fields — uploads can span months
        - Preview shows new/unchanged/changed counts with field-level diffs
        - Review step in UI with per-row accept/reject checkboxes (all accepted by default)
        - Upload accepts `accepted_change_ids` to control which changes apply
        - Unique constraint on natural key prevents DB-level duplicates

### v0.2.1: Auto-Create Categories from CSV + Category Management UI (Vertical Slice 2)
**Goal**: Categories become first-class entities — auto-created during CSV upload, manageable via a Settings UI.
**Context**: Previously, categories were only recognized if pre-seeded in the JSON fixture. New categories from Monarch CSVs showed as "unmapped" with no way to fix them in-app.
**What this unlocks**: Users can manage category groups and mappings entirely through the UI. New categories from CSV uploads are automatically tracked.
**Key tech choices**: `CategoryMapping.group_id` nullable (`None` = known but unmapped). Upload auto-creates mappings; preview reports without creating. Delete group nullifies mappings instead of deleting them.
