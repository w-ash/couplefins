### v0.3.0: Adjustment Export Engine (Vertical Slice 1)
**Goal**: Generate per-person Monarch-compatible adjustment CSVs from shared transactions for any reconciled month.
**Context**: With reconciliation working (v0.2.x), users can see who owes whom. But their individual Monarch accounts still show only what they personally paid. This version generates adjustment entries so each person's Monarch reflects their true share of shared expenses, by category.
**What this unlocks**: Complete lifecycle — the couple's Monarch accounts accurately reflect their true personal spending after importing adjustments.
**Key tech choices**: Pure domain functions for adjustment math and CSV rendering. Deterministic dedup IDs via UUID5 for Monarch re-import safety. No stored adjustment entities — adjustments are always computed fresh from current transaction data.

#### Use Cases

**UC1: "I want to export my adjustments for March"**
The user selects a month, clicks export, and gets a CSV. For every shared transaction where they were the payer, the CSV includes a credit for their partner's share (reducing their apparent category spend). For every shared transaction where their partner was the payer, the CSV includes a debit for their own share (adding to their apparent category spend). The result: after importing into Monarch, their spending-by-category is accurate.

**UC2: "I accidentally imported the same CSV twice into Monarch"**
Each adjustment row's Notes field contains a deterministic dedup ID (e.g., `[cf:a1b2c3d4e5f6]`). The user can search Monarch Notes for `cf:` to find and delete duplicates. The dedup ID is derived from the source transaction ID + target person ID, so it's stable across re-exports of the same data.

**UC3: "My partner re-uploaded their CSV — I need fresh adjustments"**
Re-uploading source transactions replaces them (new transaction IDs). Re-exporting now produces different dedup IDs (correct — source data changed). The user should delete old `cf:` adjustments in Monarch and re-import the new CSV.

**UC4: "I want to re-export and verify nothing changed"**
If no source data has changed, re-exporting produces a byte-identical CSV (same rows, same sort order, same dedup IDs). This is the core idempotency guarantee.

#### Idempotency Design

| Operation | Mechanism | Guarantee |
|-----------|-----------|-----------|
| Import (Monarch → Couplefins) | `delete_by_upload_id` + re-insert | Re-upload replaces all transactions cleanly |
| Export computation | Pure function, deterministic sort | Same input → identical output |
| Export dedup IDs | `uuid5(namespace, tx_id:person_id)` | Same source data → same IDs; changed source → changed IDs |
| Monarch re-import | `[cf:xxxx]` in Notes field | User can search Notes for `cf:` to find/delete duplicates |

#### Adjustment Engine Epic

- [ ] **Implement adjustment calculation domain logic**
    - Effort: M
    - Status: Not Started
    - What: Pure function `compute_adjustments(shared_transactions, target_person_id) → list[AdjustmentRow]` that computes per-person adjustment entries from shared transactions
    - Why: Core business logic — must be pure, deterministic, and thoroughly tested
    - Dependencies: v0.1.0 (domain entities)
    - Notes:
        - New value object `AdjustmentRow` (frozen dataclass): `source_transaction_id`, `target_person_id`, `date`, `merchant`, `category`, `amount` (Decimal), `dedup_id` (str)
        - File: `src/domain/export/adjustment_calculator.py`
        - Math per shared transaction:
            - `other_share = abs(tx.amount) × (100 - tx.payer_percentage) / 100`
            - Skip if `other_share == 0` (payer_percentage == 100)
            - For the payer: credit (positive) for expenses, debit (negative) for refunds
            - For the non-payer: debit (negative) for expenses, credit (positive) for refunds
        - Rounding: `Decimal.quantize("0.01")` with ROUND_HALF_UP
        - Deterministic sort: by `(date, merchant, category, source_transaction_id)` — total ordering
        - Dedup ID: `uuid5(COUPLEFINS_NAMESPACE, f"{tx.id}:{target_person_id}")` truncated to short hex string with `cf:` prefix. Deterministic from source data.
